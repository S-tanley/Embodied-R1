## 四、关键问题深度分析

### 4.1 为什么 overall 进入平台期？

**根本原因**: `spatial_qa` 已经饱和，而 `general_qa` 提升太慢。

```
overall = 0.1 × format + 0.9 × accuracy
```

| 来源 | 当前值 | 提升空间 | 状态 |
|------|--------|---------|------|
| format | 1.0 | 0 | 已封顶 |
| spatial_qa | ~0.81 | 很小 | 饱和 |
| general_qa | ~0.55 | 较大 | 瓶颈 |

**general_qa 为什么上不去？**

1. **batch size 太小** (`global_batch_size=32`) → GRPO 的 group baseline variance 大 → 信号噪声比低
2. **ViRL 数据更难** → 通用视觉推理比空间关系推理复杂度高
3. **general_qa 波动极大** (recent std ≈ 0.15, range 0.31-0.84) → 小 batch 导致 reward 不稳定

### 4.2 entropy 上升意味着什么？

从早期 0.92 → 近期 1.05，per-token entropy **上升了 14%**。

这不太寻常。正常 RL 训练中 entropy 应该**下降**（策略收敛）。但这里 entropy 上升，说明：

- 模型在 `<answer>` 部分变得越来越"犹豫"
- format 已完全掌握，模型把不确定性转移到了 answer 内容上
- 但 accuracy 没有同步提升 → 这种探索是**无效的**
- **结论**: 模型在"知道怎么回答"和"不知道回答什么"之间摇摆

这不是训练事故，但说明 **进一步训练可能无法带来 accuracy 提升**，只是在 answer space 里做无效探索。

### 4.3 为什么每步时间增加了 54%？

| 组件 | Step 100 | Step 500 | Step 1000 | 增幅 |
|------|----------|----------|-----------|------|
| gen (rollout) | 26.7s | 38.5s | 50.4s | **+89%** |
| old (actor fwd) | 18.0s | 32.8s | 45.0s | **+150%** |
| ref (ref fwd) | 18.7s | 18.4s | 18.4s | -2% |
| update_actor | 167s | 201s | 223s | **+33%** |
| **纯训练合计** | **230s** | **291s** | **338s** | **+47%** |
| validation (每100步) | 380s | 572s | 626s | **+65%** |

**根本原因**: `response_length` 从 112 → 150+ tokens。

vLLM 生成时间、actor forward 时间、backward 时间都与序列长度成正比。response length 增加是因为模型学会了生成更长的 reasoning 过程。

**这不是坏事**，但意味着：
- 跑到 1583 步实际需要的时间比最初估算多 50%
- 如果继续跑，时间还会进一步增加

### 4.4 Validation 开销过大

`val_freq=100`，每 100 步做一次验证。但验证时间高达 **380-626 秒**，占 step 总时间的 **50-60%**。

验证时模型要在验证集上也做 vLLM rollout 生成，而验证集样本也不少（Whatsup 138 + SAT 4000 = 4138 条）。这导致：
- 每 100 步中，有 1 步花 1000s，其他 99 步花 ~350s
- 平均下来每步实际时间 ≈ (99×350 + 1000)/100 = **353s**

**建议**: 正式训练时把 `val_freq` 从 100 改成 200-300，可以节省 15-20% 总时间。

### 7.2 对于正式 Stage 1 训练

这次 smoke test 暴露了配置问题，正式训练需要调整：

| 参数 | Smoke Test | 建议正式值 | 理由 |
|------|-----------|-----------|------|
| `global_batch_size` | 32 | **128+** | 稳定 general_qa，减少 variance |
| `micro_batch_size_per_device_for_update` | 2 | **16-32** | 提高 MFU（14%→40%+），减少 gradient accumulation |
| `micro_batch_size_per_device_for_experience` | 8 | **32-64** | 加速 old_log_probs 计算 |
| `trainer.n_gpus_per_node` | 2 | **6（全部）** | 加速 + 增大有效 batch |
| `val_freq` | 100 | **200-300** | 减少 validation 开销（占 50%+ 时间） |
| `trainer.total_episodes` | 1 | **1-2** | 1 episode 对 3B 可能足够，7B 建议 2 |
