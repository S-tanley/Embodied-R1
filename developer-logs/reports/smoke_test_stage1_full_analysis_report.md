## 4. 模型行为分析

### 4.1 策略熵——**关键异常**

正常情况下，熵损失应随策略趋于置信而**下降**。本次运行中，熵**单调上升**：

| 阶段 | 均值熵 | 标准差 | 趋势 |
|------|--------|--------|------|
| 1–100 | 0.8041 | ±0.1351 | 基准线 |
| 400–499 | 0.9298 | ±0.0571 | ↑ +15.6% |
| 800–899 | 1.0244 | ±0.0635 | ↑ +27.4% |
| 1200–1299 | 1.1770 | ±0.0694 | ↑ +46.4% |

**根因分析：**

veRL 中的熵损失衡量**策略 token 分布的不确定性**。正常 RL 训练中，熵应随模型学会哪些 token 能带来高奖励而降低。本次熵上升表明：

1. **小 batch 效应（global_batch=32）**：每次更新仅 32 个样本，梯度估计噪声大。batch 太小，策略无法从噪声中区分信号，无法自信地锁定高奖励的 token 集合。

2. **n=8 rollout，temperature=1.0**：每个 prompt 仅 8 条 rollout，探索有限。策略可能在不同答案模式之间振荡，而非收敛。

3. **回复长度增长 → 不确定的 token 更多**：回复从 131 增长到 178 tokens，模型需要做出不确定决策的位置更多，抬高了每 token 平均熵。

4. **未配置熵奖励/惩罚**：默认 veRL 配置除隐式 KL 约束外，不施加显式熵正则化。

**与回复长度的相关性：**

| 回复长度区间 | 均值熵 | 样本数 |
|------------|--------|--------|
| 100–120 tokens | 0.849 | 152 |
| 120–140 tokens | 0.987 | 397 |
| 140–160 tokens | 1.030 | 510 |
| 160–180 tokens | 1.047 | 337 |
| 180–250 tokens | 1.055 | 137 |

强正相关（r ≈ 0.6）：回复越长 → 熵越高。说明模型**愈发冗长却愈发犹豫**。

**影响**：熵越高，模型对答案越不自信，体现为：
- 输出变异性增大（推理确定性下降）
- 更容易生成格式错误或无关内容
- 推理速度变慢（需要采样更多 token）

**建议**：正式训练时将 `global_batch_size` 提升至 128+，`micro_batch_update` 提升至 16–32。可考虑添加小熵系数（如 `entropy_coeff=0.01`）来显式控制探索行为。

**MFU 均值仅 14.3%，严重偏低。**

在 H200 GPU 上训练 3B 模型，正常应达到 **35–50% MFU**。14% 的数字说明严重未充分利用算力。

**根因：**
1. **`micro_batch_size_per_device_for_update=2`**：这是主要因素。每设备每步仅 2 个样本，GPU 大量时间花在 kernel 启动开销上而非实际计算。
2. **TP=1、2 卡**：禁用了张量并行，每卡持有完整模型但处理极小的 batch。3B 模型在单张 H200 上完全放得下，但 batch size 不足以填满 GPU。
3. **FSDP 开销**：TP=1、2 卡配置下，FSDP 切片优化器状态，但计算量极小，通信/同步开销相对较大。

**MFU 应有水平：**
- `micro_batch=16`：约 30–35% MFU
- `micro_batch=32`：约 40–45% MFU
- 官方配置（8 卡，micro_batch=4）：约 20–25% MFU

**建议**：正式训练在 8× H200 上将 `micro_batch_size_per_device_for_update` 提升至 16–32，预期效果：
- MFU 从 14% 提升至 35–45%
- 总训练时间缩短约 60–70%
- 梯度质量提升（更大 batch = 更低方差）
- 可能同时修复熵异常（梯度更稳定）

### 8.2 正式训练时间估算

**当前 smoke test**：2 卡、MFU=14%，1,583 步约 10 小时

**正式训练预测**（8 卡、batch=128、micro=16、TP=2、15 episodes）：
- 预计总步数：约 24,000（smoke test 的 15 倍）
- MFU 提升（14% → 35%）：**每步快约 2.5 倍**
- GPU 增加 4 倍：**快约 4 倍**（含 TP 与通信开销）
- 综合加速：**约 10 倍**
- **预计正式训练时长**：10h × 15 / 10 = **约 15 小时**

若 micro_batch 提升至 32，MFU 可达 45%，总时间可进一步降至 **10–12 小时**。

### 8.3 架构层面调整

1. **熵正则化**：考虑添加熵系数调度策略：
   - 训练开始：`entropy_coeff=0.02`（鼓励探索）
   - 训练中期：退火至 `0.005`
   - 可防止熵异常，并促进模型输出更置信的答案

2. **回复长度控制**：对超过 200 tokens 的回复添加软惩罚，或硬性上限设为 256。140–160 token 范围是奖励最优区间。

3. **梯度裁剪**：将裁剪阈值降至 5.0，防止类似 step 1373 的后期尖峰。

4. **学习率调度**：当前固定 LR=1e-6 可用，但可考虑：
   - 前 5% 步做 warmup
   - 最后 20% 做 cosine 衰减，在平台期稳定收敛

5. **数据重采样**：通用 QA 是瓶颈，若数据混合比例可配置，建议：
   - 上采样通用 QA 样本
   - 引入更难的通用 QA 样本
   - 或若空间 QA 已充分饱和，适当降低其权重

### 8.4 正式训练实时监控清单

正式训练过程中重点关注以下指标：

- [ ] **综合奖励**——应稳步上升至约 0.85–0.90 后趋于平台
- [ ] **空间 QA 精度**——预计 step 5,000–8,000 前后饱和，约 0.85
- [ ] **通用 QA 精度**——Stage 1 结束时目标 >0.65
- [ ] **熵损失**——应**下降**或趋于平稳；若上升超过 1.0，检查 batch size
- [ ] **回复长度**——应稳定在 200 tokens 以内；若超过 220，添加长度惩罚
- [ ] **梯度范数**——若连续 3 步以上 >10，触发告警
- [ ] **MFU**——应在 30–45%；若 <25%，触发告警
- [ ] **单步耗时**——应保持稳定；若增长超过 30%，说明回复长度可能出现爆炸
- [ ] **显存占用**——应 <80 GB；若超过 100 GB，触发告警

## 9. 结论

`smoke_test_stage1` 成功验证了 Embodied-R1 训练流程在 2× H200 GPU 上的可用性。模型在空间推理任务上展现出**强劲的学习能力**，峰值综合奖励达到 **0.919**，格式合规性也高度稳定。

但在扩展至正式训练之前，有三个问题需要处理：

1. **熵异常**（上升 46%）：策略未能自信收敛，很可能是极小的 batch size（32）和低 MFU（14%）共同造成的。
2. **通用 QA 瓶颈**（卡在约 0.55）：无论空间 QA 表现多好，通用 QA 的低分始终压制整体奖励的提升上限。
3. **验证开销**：浪费了约 10% 的总训练时间。

实施建议修复后——将 `micro_batch_update` 提升至 16–32、`val_freq` 提升至 200–300，并酌情调整熵正则化——预期正式 Stage 1 训练将：
- 在 8× H200 上约 **10–15 小时**完成（对比 smoke test 设置下约 150 小时）
- 达到 **35–45% MFU**
- 综合奖励稳定在 **0.85–0.90**
- 输出置信、格式规范，熵值 <0.90

`workdir/smoke_test_stage1/global_step_1000/` 的 checkpoint 可用于检查，并作为与正式训练结果对比的基准。
